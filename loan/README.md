# Loan Marketplace

## Prompt

Help me refine a PRD about a loan marketplace.

The actors in this marketplace are:

- lender: a person that have money and is willing to lend it to borrowers.
- borrower: a person who will provide guarantees to get loans

Be Clear and Concise: Avoid jargon and ambiguity. Use bullet points, tables, or visuals for clarity.
Make It Collaborative: Involve engineering, design, and other teams early to ensure feasibility and alignment.
Focus on “Why” and “What,” Not “How”: Leave implementation details to the engineering team unless specific technical constraints are critical.

# Product Requirements Document: Loan Marketplace

## 1. Introduction

- Purpose: Explain why the product or feature is being built. What problem does it solve, and for whom?
- Objectives: Define the high-level goals (e.g., improve user engagement, increase revenue, streamline operations).
- Scope: Outline what’s included and explicitly state what’s out of scope to manage expectations.
- Target Audience: Specify the primary users or customers (e.g., personas, demographics, or user types).

## 2. Background

- Problem Statement: Clearly articulate the problem or opportunity the product/feature addresses.
- Market or User Research: Summarize insights from user interviews, surveys, or market analysis supporting the need.
- Competitive Analysis: Highlight competitors and how the product differentiates itself.

## 3. Requirements

- Functional Requirements: Detail the core features and functionalities. Use clear, specific language (e.g., “The app must allow users to upload images up to 10MB in JPG or PNG format”).
- Non-Functional Requirements: Specify performance, scalability, security, accessibility, or usability standards (e.g., “The system must handle 10,000 concurrent users with <2s response time”).
- User Stories/Use Cases: Write user-centric stories (e.g., “As a [user type], I want [action] so that [benefit]”). Include acceptance criteria for each.
- Constraints: Note technical, budgetary, or time limitations (e.g., “Must integrate with existing AWS infrastructure”).

## 4. Design & UX

- Wireframes/Mockups: Include visual references or links to designs (if available) to clarify the user interface.
- User Flow: Describe how users will interact with the product/feature step-by-step.
- Accessibility Requirements: Ensure compliance with standards like WCAG (e.g., “Support screen readers for visually impaired users”).

## 5. Technical Specs

- Architecture Overview: High-level description of the technical approach (e.g., APIs, databases, or frameworks).
- Dependencies: List integrations, third-party services, or existing systems required.
- Data Requirements: Define data inputs, outputs, storage, or privacy considerations (e.g., GDPR compliance).
- Performance Metrics: Specify expected performance benchmarks (e.g., uptime, latency).

## 6. Success Metrics

- Key Performance Indicators (KPIs): Define measurable outcomes (e.g., “Increase user retention by 15%” or “Reduce checkout time by 20%”).
- Analytics: Specify how success will be tracked (e.g., Google Analytics, custom dashboards).
- Validation Plan: Outline how you’ll test whether the product meets its goals (e.g., A/B testing, user feedback).

## 7. Timeline

- Release Plan: Provide a high-level timeline with key milestones (e.g., design review, alpha release, beta testing).
- Prioritization: Highlight must-have vs. nice-to-have features (e.g., using MoSCoW: Must have, Should have, Could have, Won’t have).
- Dependencies: Note any external factors affecting the timeline (e.g., third-party vendor deliverables).

## 8. Stakeholders

- Team: List key stakeholders (e.g., product manager, designers, developers) and their responsibilities.
- Approval Process: Define who needs to sign off on the PRD or specific deliverables.

## 9. Risks & Assumptions

- Assumptions: Document assumptions made during planning (e.g., “Users have stable internet access”).
- Risks: Identify potential challenges and mitigation plans (e.g., “Risk: API delays; Mitigation: Build mock APIs for testing”).

## 10. Appendices

- Glossary: Define technical terms or jargon for clarity.
- References: Include links to research, competitor analyses, or related documents.
- FAQs: Address common questions from stakeholders to preempt confusion.
